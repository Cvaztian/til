● Codigo C/C++ y Programacion CRITICA

▶
Segun resenas: Te dan codigo y debes predecir output. Tambien preguntas sobre tipos de datos y escribir codigo. JD pide 1-3 anos en C/C++ embedded.

🔴 Predecir Output de Codigo C

Trampas: precedencia de operadores, signed vs unsigned, integer promotion, overflow, short-circuit, post vs pre increment, punteros, sizeof arrays vs punteros, null terminator.
// TRAMPA 1: signed vs unsigned unsigned int a = 0; int b = -1; if (b < a) printf("menor"); else printf("mayor o igual"); // Output: "mayor o igual" // b se promueve a unsigned → 0xFFFFFFFF > 0 // TRAMPA 2: sizeof char s[] = "hello"; printf("%lu\n", sizeof(s)); // 6 (incluye '\0') char *p = "hello"; printf("%lu\n", sizeof(p)); // 4 u 8 (puntero!) // TRAMPA 3: precedencia int x = 2; printf("%d\n", x & 1 == 0); // == > & en precedencia // x & (1==0) → x & 0 → 0 (no lo que esperabas)
signed vs unsigned: -1 > 0u en C. Integer promotion es trampa #1.

🔴 Tipos de Datos Primitivos vs Compuestos

Primitivos: int, char, float, double, _Bool, void.
Compuestos: arrays, structs, unions, enums, punteros.
Embedded: uint8_t, int16_t, uint32_t (stdint.h, tamano fijo).
sizeof ARM 32-bit: char=1, short=2, int=4, long=4, float=4, double=8, pointer=4.
// struct padding struct A { char a; int b; char c; }; // sizeof=12 (padding!) struct B { int b; char a; char c; }; // sizeof=8 (reordenado) // union: todos comparten memoria, sizeof = miembro mas grande
sizeof(struct) != suma de miembros. union comparte memoria.

🔴 volatile — Pregunta #1 en entrevistas embedded

Que hace: Cada lectura/escritura → acceso real a memoria. Prohibe al compilador cachear/optimizar.
Cuando: Registros HW, variables de ISR, variables RTOS, registros read-to-clear.
Que NO hace: No es atomico. No garantiza ordenamiento.
// Sin volatile → compilador puede optimizar a un solo read while (GPIOA->IDR & (1<<0)) {} // posible loop infinito // Con volatile → lee del hardware cada vez volatile uint32_t *reg = (volatile uint32_t *)0x40020010; while (*reg & (1<<0)) {} // correcto

🔴 Escribir Codigo: Patrones Comunes

Clasicos: reverse string, palindrome, binary search, bit manipulation, FizzBuzz, swap sin temp, linked list, find max/min.
// Bitwise — diario en embedded // SET bit N: reg |= (1 << N) // CLR bit N: reg &= ~(1 << N) // TOGGLE bit N: reg ^= (1 << N) // CHECK bit N: if (reg & (1 << N)) void reverse(char *s) { int len = strlen(s); for (int i = 0; i < len/2; i++) { char tmp = s[i]; s[i] = s[len-1-i]; s[len-1-i] = tmp; } } // Power of 2: n && !(n & (n-1))

🟠 Punteros, Stack vs Heap

Stack: automatico, rapido, limitado (~kB en embedded). Heap: malloc/free, fragmentacion, PROHIBIDO en DO-178C DAL-A.
Punteros: int *p = &x; *p dereference; p++ avanza sizeof(int) bytes.
Function pointers: void (*fp)(int) = &func; — callbacks de ISR.
Stack overflow en embedded es silencioso (sin MMU). malloc prohibido en avionica certificada.

● TDD, Verificacion y Certificacion DESDE JOB DESC CRITICA

▶
El JD menciona TDD explicitamente y dedica 4 de 9 responsabilidades a testing/verification/documentation. Esto es la mitad del trabajo diario.

🔴 TDD (Test-Driven Development)

Ciclo: Red (test que falla) → Green (codigo minimo que pasa) → Refactor (limpiar sin romper tests).
En embedded: Compilar para host (PC), mockear hardware, correr tests con Unity/CppUTest.
En avionica: Cada test verifica un requisito. Sin test = sin certificacion.
// Ejemplo TDD // PASO 1 Red: test primero void test_checksum_empty(void) { uint8_t buf[] = {}; TEST_ASSERT_EQUAL(0, calc_checksum(buf, 0)); } // PASO 2 Green: implementacion minima uint8_t calc_checksum(uint8_t *buf, size_t len) { uint8_t sum = 0; for (size_t i = 0; i < len; i++) sum ^= buf[i]; return sum; } // PASO 3: Refactor si necesario

🔴 DO-178C — Certificacion Software Aeronautico

Que es: Estandar internacional para SW aeronautico. Joby/Aeronautical SW lo siguen para FAA.
DAL: A=catastrofico (MC/DC), B=peligroso (decision cov), C=mayor (statement cov), D=menor, E=sin efecto.
Conceptos clave: Traceability bidireccional, independence (quien escribe ≠ quien verifica), MC/DC, no dynamic allocation, determinismo, particion espacial (MPU) y temporal.

🔴 MISRA C — Subconjunto Seguro de C

Reglas principales: No malloc/free, no recursion, no goto, siempre usar {} en if/else, no casting implicito peligroso, siempre inicializar variables, no side effects en argumentos (x++ en args = prohibido).

🟠 Test Coverage

Statement: cada linea ejecutada. Decision/Branch: cada if true y false. MC/DC: cada condicion individual afecta resultado independientemente (DAL-A).
// MC/DC para: if (A && B) // Test 1: A=T, B=T → true (baseline) // Test 2: A=F, B=T → false (A cambio resultado) // Test 3: A=T, B=F → false (B cambio resultado) // Solo 3 tests, no 4.

🟠 Requirements Writing

Formato: "The system shall [accion] [condicion] [criterio]." Cada req debe ser verificable y no ambiguo.
Ej: "The system shall calculate motor temperature at 100 Hz ± 1%."
Traceability: System Req → SW Req → Design → Code → Test → Result. Bidireccional.

● Electronica, Circuitos y Hardware ALTA

▶

🔴 Protocolos: SPI vs I2C vs UART vs CAN

UART: Asincrono, 2 cables TX/RX, 115200, frame 8N1, punto a punto. Debug.
SPI: Sincrono SCLK, 3+N cables, full-duplex, ~50MHz. Flash, ADC, IMU.
I2C: Sincrono SCL, 2+pull-ups, half-duplex, 7-bit addr, ACK/NACK, 400kHz.
CAN: Diferencial, multi-master, arbitraje por ID, 120Ω term, 1Mbps. ARINC 825.
Mas rapido=SPI. Menos cables=I2C. Mas robusto/avionica=CAN. I2C necesita pull-ups.

🟠 ADC/DAC

LSB = V_ref / 2^N. 12-bit 3.3V → LSB=0.806mV. Max=4095 (no 4096).
Nyquist: f_s ≥ 2*f_max. Aliasing = IRREVERSIBLE. Anti-aliasing obligatorio.
SAR=tu Cortex-M. Sigma-Delta=precision. Flash=ultra-rapido.

🟠 ARM Cortex-M Arquitectura

Registros: R0-R3 args, R4-R11 callee-saved, R13=SP, R14=LR, R15=PC.
Harvard modificado. Pipeline 3 etapas (Cortex-M3/M4). NVIC 240 IRQs.
Cortex-M3/M4 = 3 etapas (no 5). STM32F4 NO tiene data cache.

🟡 Logica Digital + Transistores

Compuertas, DeMorgan, K-maps, complemento a 2, flip-flops, setup/hold.
BJT switch: saturacion=ON, β_forzado=10. MOSFET: V_GS>V_th=ON, R_DS(on) mΩ.
"Saturacion" = ON en BJT = amplificador en MOSFET (opuestos).

● Embedded Systems & RTOS ALTA

▶

🔴 RTOS Conceptos (Desired en JD)

FreeRTOS: Tareas con stack propio (PSP), scheduler preemptive por prioridad.
Sincronizacion: Semaforo (contador), Mutex (owner + priority inheritance), Queue (datos).
Priority inversion: Tarea alta bloqueada por baja. Fix: priority inheritance en mutex.
Context switch: PendSV. SysTick = tick (1ms).
Mutex ≠ Semaforo. Mutex tiene owner. Semaforo es solo contador.

🟠 Memory-Mapped I/O + Interrupts + DMA

No hay I/O especial en ARM. Todo volatile + direccion. Siempre read-modify-write para bits.
Polling: simple, desperdicia CPU. Interrupts: hardware avisa, CPU libre. DMA: sin CPU.
#define GPIOA_ODR (*((volatile uint32_t *)0x40020014)) GPIOA_ODR |= (1 << 5); // SET bit 5 GPIOA_ODR &= ~(1 << 5); // CLR bit 5
