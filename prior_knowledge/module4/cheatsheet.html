Von Neumann vs Harvard entrevista

Von Neumann

Buses:1 compartido (instrucciones + datos)
Problema:Von Neumann bottleneck — solo una cosa a la vez
Usado en:x86 PCs, sistemas propósito general

Harvard

Buses:2 separados — instrucciones Y datos en paralelo
Ventaja:Doble ancho de banda efectivo
Usado en:DSPs, AVR, PIC

Modified Harvard — ARM Cortex-M ⭐

Buses:Múltiples internos, un solo espacio de addr
I-Codeinstrucciones desde Flash
D-Codeconstantes/datos desde Flash
SystemSRAM, periféricos, resto
DMAindependiente del core
DMA transfiere por System bus mientras core ejecuta por I-Code → simultáneo, sin espera

Pipeline & Instrucciones
Pipeline entrevistacálculo

5 Etapas Clásico (concepto)

I1: ADD R2, R0, R1IFIDEXMEMWB
I2: SUB R4, R2, R3IFstallstallIDEX
I3: AND R5, R6, R7IFstallstallID
⚠ RAW hazard: I2 necesita R2 que I1 no terminó de escribir → 2 stalls sin forwarding
Con forwarding EX→EX: 0 stalls. Load-use: 1 stall inevitable (dato llega en MEM).

Cortex-M3/M4: 3 etapas

Etapas:Fetch → Decode → Execute
⚠ PC durante ejecución = addr_actual + 4 (2 instrucciones adelante, herencia de 3 etapas)
Branch penalty:≈ 3–4 ciclos (vs ~15 en desktop)

Los 3 Tipos de Hazard entrevista

TipoCausaSolución
Data (RAW) I2 lee lo que I1 escribe Forwarding, reorder, NOP
Control Branch cambia PC Branch prediction, flush, unroll
Structural Dos instrs. necesitan mismo HW Cache I/D separada, más unidades

Latencias de Instrucciones (Cortex-M4)

ADD, SUB, AND, MOV1 ciclo
LDR (memory)2 ciclos
MUL1–3 ciclos
LDM de N registrosN+1 ciclos
Branch tomado3–4 ciclos penalty
Tail-chain ISR6 ciclos (vs 12 normal)
CPI_real = 1 + stalls. Speedup = N etapas (teórico). Compilador reordena instrucciones para minimizar RAW.

En STM32F4 (Cortex-M4)

ART Accelerator = prefetch buffer para instrucciones. Sin cache de datos. SRAM: 0 wait states.

En STM32H7 (Cortex-M7)

L1 cache de instrucciones Y datos (típicamente 16–32 KB c/u). Write-back cache → problema con DMA.

Localidad de Referencia

Temporal:accediste a X → pronto lo accedés de nuevo (loops)
Espacial:accediste a X → pronto accedés X±n (arrays, código)
⚠ Recorrido column-major en C = cache thrashing. Usar row-major: A[i][j] con el índice j variando rápido.

Campos de la dirección

Dirección: [ Tag | Index | Offset ]
Offsetlog₂(bytes por línea) bits
Indexlog₂(capacidad / vías / tamaño_línea) bits
Tag32 - Index_bits - Offset_bits

Ejemplo de Cálculo ⭐

; Cache: 4 KB, líneas 32 bytes, direct-mapped Offset = log₂(32) = 5 bits → [4:0] Sets = 4096 / 32 = 128 Index = log₂(128) = 7 bits → [11:5] Tag = 32 - 7 - 5 = 20 bits → [31:12] ; ¿Cuántas vías tiene esta cache? ; direct-mapped = 1-way (1 línea por set) ; Si fuera 2-way: sets = 4096 / (32 × 2) = 64 ; index = log₂(64) = 6 bits

Trampas Comunes trampa
Error comúnCorrección
"PC apunta a instrucción actual" PC = addr + 4 en Cortex-M (3 etapas)
"Bit 0 del PC no importa" Bit 0 = 1 siempre (Thumb). Branch a par → HardFault
"Write-back siempre es mejor" Con DMA: incoherencia. Flush antes del DMA.
"Direct-mapped es más rápido" Thrashing → miss rate 100%. Set-assoc balancea mejor.
"Cortex-M4 tiene 5 etapas" M3/M4 tiene 3 etapas. 5 etapas es el concepto clásico.
"Cortex-M4 tiene data cache" STM32F4 (M4): NO. STM32H7 (M7): SÍ.
"LDR no causa hazards" Load-use hazard = 1 stall inevitable. Dato en MEM no en EX.
"R13 siempre es el mismo stack" MSP (OS/IRQ) vs PSP (cada tarea). CONTROL[0] selecciona.
"Timer APB1 = 42 MHz" Si prescaler APB1 > 1: timer = APB1 × 2 = 84 MHz.

Relevancia para el Puesto — Aeronautical SW / Joby aviónica
Modified Harvard
DMA transfiere por System bus mientras core ejecuta por I-Code bus — paralelo, sin espera.

ARM Thumb-2 + AAPCS
Leerás ensamblador al debuggear con GDB/JTAG. AAPCS te dice qué registros mirar para args y return values.
Pipeline hazards
El compilador los evita, pero si escribís ASM crítico o analizás disassembly, necesitás reconocer load-use stalls.

Cache + DMA coherency
Bug sutil en STM32H7: solo aparece en hardware, no en simulación. Flush es obligatorio antes de DMA con write-back cache.
Privilege + MPU
DO-178C requiere partición espacial. Configurarás la MPU en el RTOS de producción para aislar tareas críticas.

Vector table + Boot
Al debuggear un HardFault, la vector table te dice qué handler saltó. El linker script + startup code son los primeros archivos que editarás en un proyecto nuevo.
